# State Space Models (Mamba)

The [Sequence Models](/dl-anatomy/architectures/sequence-models-en) article ended at a fork: RNNs/LSTMs keep a **compressed state** but are hard to parallelize across time; Transformers remove recurrence and train in parallel, but standard attention costs $O(n^2)$ and the KV cache grows with context length.

**State Space Models (SSMs)** take a third route: keep the idea of state, but organize it as a structured linear system. The goal is not to "return to RNNs" in the old sense, but to build a sequence modeling block with compact inference memory and efficient training on long sequences.

## The problem: precise memory or compact memory?

A Transformer stores the past as a **memory bank**: every token leaves its own key/value entry in the KV cache. When the current token needs information, it can attend directly to old tokens. This is a powerful retrieval mechanism, but memory grows with context length.

An RNN does the opposite: the past is compressed into a fixed state. Inference is compact, but training is hard to parallelize because step $t$ depends on the state from step $t-1$.

SSMs try to keep the best of both:

- fixed state like an RNN at inference time
- parallel computation like convolution at training time
- enough linear structure to compute long kernels efficiently

## Three views of an SSM

A modern SSM has three views:

| View | Role |
|---|---|
| Continuous view | models a dynamical system in continuous time |
| Recurrent view | runs one step at a time with fixed state, useful for inference/streaming |
| Convolution view | applies a long kernel over the whole sequence, useful for parallel training |

This is the central insight: an SSM is not just a "linear RNN." It is a layer whose computation schedule can change depending on the stage.

## Continuous view — from dynamical systems

SSMs come from control theory. We start with a continuous-time system:

$$
\frac{dh(t)}{dt} = A h(t) + B x(t), \qquad y(t) = C h(t)
$$

Where:

- $x(t)$ is the input entering the system
- $h(t)$ is the hidden state, a compressed history
- $y(t)$ is the output read from the state
- $A$ controls how the state evolves and "forgets"
- $B$ controls how input is written into the state
- $C$ controls which part of the state is read out

In the full control-theory form, the output can also include a direct input path:

$$
y(t) = C h(t) + D x(t)
$$

The term $D x(t)$ lets the input affect the output without passing through state. In deep learning SSMs, this is often treated like an easily computed skip connection, so this article ignores $D$ and focuses on the important part: **state dynamics**.

## Discretization — from continuous time to tokens

Deep learning data usually arrives as discrete sequences: tokens, audio frames, time-series steps. So the continuous system must be **discretized** into:

$$
h_t = \bar{A} h_{t-1} + \bar{B} x_t, \qquad y_t = C h_t
$$

The notation $\bar{A}, \bar{B}$ reminds us that these are the discretized versions of $A, B$. Discretization depends on the step size $\Delta$: how much continuous time one discrete step represents.

Intuitively, a larger $\Delta$ makes state change more strongly between steps; a smaller $\Delta$ makes state evolve more smoothly. This is why $\Delta$ becomes important in Mamba: the model can learn different update speeds for different tokens.

<Note type="intuition">
If an RNN learns a free nonlinear update, an SSM learns structured dynamics: how state should decay, preserve, and write information. That linear structure is what makes parallel computation possible.
</Note>

## Recurrent view — inference like an RNN

The discrete equation looks very close to an RNN:

$$
h_t = \bar{A} h_{t-1} + \bar{B} x_t
$$

At inference time, we only need to keep the current state $h_t$. A new token arrives, the state updates, the output is read, and the old token can be discarded. Memory does not grow with context length as it does with a KV cache.

The difference from a classic RNN is that the core state update is linear in the state and input. Nonlinearities usually live in projections and gates around the block, not directly inside the core recurrence.

## Convolution view — parallel training

With a linear recurrence, we can unroll a few steps:

$$
h_1 = \bar{B} x_1
$$

$$
h_2 = \bar{A}\bar{B}x_1 + \bar{B}x_2
$$

$$
h_3 = \bar{A}^2\bar{B}x_1 + \bar{A}\bar{B}x_2 + \bar{B}x_3
$$

Reading the output gives:

$$
y_t = C h_t = \sum_{k=0}^{t-1} C\bar{A}^{k}\bar{B}x_{t-k}
$$

In other words, the output is a **convolution** between the input and a long kernel:

$$
y = x * \bar{K}, \qquad \bar{K} = (C\bar{B},\; C\bar{A}\bar{B},\; C\bar{A}^2\bar{B},\; \dots)
$$

This is the most elegant part of SSMs: the same model has two computation schedules.

| Schedule | Used when | Meaning |
|---|---|---|
| Convolution / kernel | Training | process the whole sequence in parallel |
| Recurrence / state update | Inference | run token by token with fixed state |

With S4 and other structured SSMs, the kernel can be computed and applied very efficiently, often near-linear in sequence length in modern implementations. With Mamba, selective scan is designed to run linearly in sequence length.

## How are SSMs different from RNNs and Transformers?

RNNs, Transformers, and SSMs all answer the same question: **how should the current token use the past?**

| Architecture | Past memory | Cost with sequence length | Strength | Weakness |
|---|---|---|---|---|
| RNN/LSTM | fixed state | $O(n)$ but sequential | compact inference, natural streaming | hard to parallelize, long dependencies are difficult |
| Transformer | KV cache / attention to every token | $O(n^2)$ for standard attention | flexible content lookup | memory and compute grow quickly |
| SSM/Mamba | structured fixed state | linear or near-linear in sequence length | long sequences, streaming, compact inference | compressed state may lose details that need exact lookup |

<Note type="warning">
A fixed state is not "free." It is an intentional bottleneck. If a task needs exact retrieval of a distant detail, attention still has a natural advantage because it can look directly at old tokens.
</Note>

## S4 — choosing $A$ for long memory

If we only write the linear formula, the big question remains: how should we choose $A$ so the state is stable and still remembers long dependencies?

Matrix $A$ controls how state decays and preserves history. If $A$ is chosen poorly, state may forget too quickly or become unstable. **S4 (Structured State Space Sequence Model)** answers this by imposing special structure on the state matrix, related to HiPPO, a way to compress continuous history into state while preserving long-range information.

At the architecture level, the important points are:

- S4 shows that an SSM is not just a simple linear RNN.
- State can be designed to preserve long history in a disciplined way.
- With the right structure, SSMs can compete on very long sequences where standard attention is expensive.

We do not need the full S4 algebra in this anatomy article. The key takeaway is: **S4 proved that a state-space layer could become a serious sequence modeling block, not just an idea borrowed from control theory.**

## Mamba — from LTI to selective state

Classic SSMs are usually **linear time-invariant (LTI)**: after training, parameters like $\bar{A}, \bar{B}, C$ are fixed over time. That means the layer uses the same dynamics for every token. But in language, code, or audio, not every token matters in the same way.

This is the weakness of fixed SSMs: they compress the past efficiently, but struggle to decide by content which token should be retained and which should be ignored.

### Selective Parameters

Mamba brings **content dependence** into SSMs. The parameters that read from and write to state depend on the input:

$$
\Delta_t = \Delta(x_t), \qquad B_t = B(x_t), \qquad C_t = C(x_t)
$$

- $\Delta_t$ controls how quickly the state updates at this token
- $B_t$ controls how the current token is written into state
- $C_t$ controls which part of state is read to produce output

Intuition: Mamba is not just one fixed filter running across the sequence. Each token helps decide **what to write, what to keep, and what to read**.

<SelectiveScanViz lang="en" />

### Selective Scan

Once $B_t, C_t, \Delta_t$ depend on the input, Mamba is no longer LTI and there is no longer one fixed convolution kernel as in classic S4. The simple convolution view breaks.

Mamba compensates with **selective scan**: a GPU-optimized parallel scan algorithm. It keeps the idea of recurrence with state, but implements it in a hardware-aware way: avoiding excessive writes of intermediate states to slow memory, using kernel fusion and recomputation so training remains efficient.

<Note type="result">
Mamba matters not just because it "does not use attention." The key is that it turns linear recurrence into a modern block: content-dependent, GPU-friendly, compact at inference, and suited to long sequences.
</Note>

## The Mamba block inside a neural network

In real models, Mamba is not used as a bare state-space equation. It is packaged as a neural network block, usually with:

1. projection from embeddings into hidden channels
2. short convolution along the sequence to mix local context
3. selective SSM / selective scan for long-range information flow
4. gating and output projection
5. residual connection and normalization around the block

This makes Mamba closer to how we think about Transformer blocks: not just one core operation, but a **module** with mixing, gating, residual paths, and normalization. The core operation differs:

| Block | Main token mixing | Memory |
|---|---|---|
| Transformer block | self-attention | per-token KV cache |
| Mamba block | selective scan | fixed state |

If a Transformer asks "which token should this token look at?", Mamba asks "how should this token update the state?"

## Limitation: state does not mean perfect state tracking

Because SSMs have state, it is tempting to assume they are automatically better than Transformers at every "stateful" task. That needs care.

The paper *The Illusion of State in State-Space Models* shows that for some formal state-tracking tasks — such as tracking permutations or exact discrete state updates — linear and Mamba-style bounded-depth SSMs do not automatically have the same expressive power as simple RNNs. In other words, "having hidden state" is not the same as "solving every stateful problem."

This does not undermine SSM/Mamba. It clarifies the trade-off:

- SSMs are strong when long sequences need compact memory and inference.
- Attention remains strong when exact content retrieval matters.
- RNNs still have a natural bias for some hard sequential state updates.

## Where SSMs stand today

SSM/Mamba should not be read as a simple "Transformer killer." It is better understood as a powerful architecture branch for problems where attention cost becomes the bottleneck.

SSMs are especially useful when:

- sequences are very long and $O(n^2)$ attention is too expensive
- streaming inference with fixed memory matters
- the data behaves like a continuous signal, such as audio, time-series, or genomics
- we want to reduce KV-cache pressure in long-context models

Attention remains very strong when:

- exact retrieval of distant details matters
- flexible matching between every pair of tokens is useful
- the workload is already deeply optimized around Transformers

That is why many practical directions use **hybrid architectures**: interleaving attention layers and SSM layers. Attention handles sharp content retrieval; SSMs carry long-range information at linear cost.

## Three views of sequence modeling

Instead of treating history as one clean line of progress, it is better to see sequence architecture as three families:

| Family | Evolution |
|---|---|
| Recurrence | RNN → LSTM/GRU |
| Attention | Seq2Seq attention → Transformer |
| State-space | S4 → Mamba → Mamba-2 / hybrid SSM |

These families do not replace one another cleanly. They are different trade-offs among **memory**, **parallelism**, **retrieval ability**, and **inference cost**.
