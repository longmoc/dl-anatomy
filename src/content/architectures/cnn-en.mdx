# Convolutional Networks

## An inductive bias for images

A fully-connected layer wires every pixel to every neuron. For a $224\times224\times3$ image, just the first layer into 1000 neurons is ~150 million parameters — and it ignores an obvious fact: **nearby pixels are more strongly correlated** than far-apart ones. Long-range relations do exist (the front and rear wheels, the two eyes of a face), but they're typically built up gradually through deeper layers. Convolution bakes in this assumption through three principles:

- **Locality**: each neuron looks at only a small region (the kernel), e.g. $3\times3$ — not the whole image
- **Weight sharing**: the same kernel slides across the image → it detects one feature (edge, corner) at *every* position
- **Translation equivariance**: if a pattern moves to a different position in the input, its response on the feature map moves the same way

<Note type="intuition">
Weight sharing is a strong form of regularization: instead of learning per-position, the model is forced to learn features that are *useful everywhere*. This is why CNNs are far more data-efficient than MLPs on images — a textbook case of "putting the right assumption into the architecture."
</Note>

## The convolution operation

A kernel (or filter) is a small matrix that slides over the input; at each position the output is a **weighted sum** of the patch it covers:

$$
(I * K)(x, y) = \sum_{i}\sum_{j} I(x-i,\, y-j)\, K(i, j)
$$

> This is the mathematical definition of convolution — the kernel is **flipped** (the $x-i$ sign). But what DL libraries actually compute is **cross-correlation**: the kernel is **not** flipped. Because the kernel is learned from data this barely matters, and we still loosely call it "convolution". The operation actually computed is:

$$
(I \star K)(x, y) = \sum_{i}\sum_{j} I(x+i,\, y+j)\, K(i, j)
$$

### Channels and feature maps

A color image has 3 **channels** (R, G, B). A conv filter isn't just $3\times3$ but $C_{in}\times3\times3$ — it spans *all* input channels and sums them into **one** feature map. Using $C_{out}$ different filters → $C_{out}$ output feature maps. The shapes (exactly the weight shape of PyTorch's `Conv2d`, $(C_{out}, C_{in}, k, k)$):

$$
\begin{aligned}
\text{input } X &\in \mathbb{R}^{C_{in} \times H \times W} \\[2pt]
\text{filter bank } W &\in \mathbb{R}^{C_{out} \times C_{in} \times k \times k} \\[2pt]
\text{output } Y &\in \mathbb{R}^{C_{out} \times H' \times W'}
\end{aligned}
$$

Each feature map is a "map" of where one specific feature appears in the image.

### 1×1 convolution

A $1\times1$ kernel mixes no spatial information (it covers a single pixel), but it **mixes information across channels**: at each position it learns a projection from $C_{in}$ to $C_{out}$. Common roles: increase/decrease the channel count, mix features across channels, and act as a **bottleneck** (in Inception, ResNet, MobileNet).

<Note type="intuition">
If a $3\times3$ conv asks *"what does the neighborhood say?"*, a $1\times1$ conv asks *"how should the channels at this one position be mixed?"*
</Note>

### Stride, padding, output size

- **Kernel size** $k$ — the field of view per step (typically 3 or 5)
- **Stride** $s$ — the step when sliding; $s>1$ downsamples
- **Padding** $p$ — a border of zeros around the input to preserve size

Output size per dimension:

$$
H' = \left\lfloor \frac{H + 2p - k}{s} \right\rfloor + 1, \quad W' = \left\lfloor \frac{W + 2p - k}{s} \right\rfloor + 1
$$

### Parameter savings

A $3\times3$ conv layer with $C_{in}=C_{out}=64$ has

$$
3 \times 3 \times 64 \times 64 = 36{,}864
$$

weights (plus $64$ biases if the layer uses bias) — **independent of image size**. In many modern CNNs, when a conv is followed by BatchNorm the bias is dropped, since BatchNorm already has its own shift parameter.

If you replaced the convolution with a fully-connected mapping over the whole feature map, the parameter count would explode because every position needs its own weights — and, more importantly, an FC layer loses both locality and weight sharing.

## Pooling and downsampling

After a few conv layers we usually **pool** to reduce resolution: split the feature map into cells and take the max (**max pooling**) or average (**average pooling**) of each. Benefits: less computation, larger receptive field, and a bit of invariance to small shifts.

In modern CNNs, downsampling **doesn't have to** use pooling — many architectures (ResNet, ConvNeXt) use a **strided convolution** ($s>1$) to both reduce resolution and *learn* how to downsample. The "conv → relu → pool" pattern is typical of classic CNNs, not mandatory for every CNN.

<Note type="theorem">
A key distinction:

- **Convolution** is fundamentally **translation-equivariant**: shift the input and the feature map shifts accordingly (ignoring border/padding effects).
- **Pooling** adds a degree of **local / approximate invariance**: a small shift *within* a pooling cell may not change the output — but a shift past the cell, or off the grid, still does.

CNNs combine both: the feature map keeps relative position information, while the final prediction becomes less sensitive to small shifts.
</Note>

## Receptive field

A neuron's **receptive field** is the input region it "sees". A single $3\times3$ conv sees only $3\times3$ pixels — very local. But stacking layers grows that field. With $L$ convolution layers of kernel $k$, **stride 1, dilation 1, no downsampling**:

$$
RF = 1 + L\,(k-1)
$$

This is the simple case; with stride, pooling, or dilation the receptive field grows differently (faster).

<ReceptiveFieldViz lang="en" />

This is the key: from **purely local** operations, a deep enough stack can grow its field of view to nearly the whole image — *in theory*.

> The formula above is the *theoretical* receptive field. In practice not every pixel in that region contributes equally — those near the center have far more influence. This actual influence is the *effective receptive field*, usually smaller than the theoretical one.

## Feature hierarchy

Because the receptive field grows with depth, features grow more abstract too:

- **Early layers**: edges, corners, color gradients (small field of view)
- **Middle layers**: textures, patterns, simple parts (eyes, wheels)
- **Late layers**: complex parts → whole objects

The CNN architecture biases the model toward learning a local-to-global hierarchy (via locality, weight sharing, downsampling, and depth), but the specific features are still learned from data — nobody programs "this is an edge, that is an eye."

## The classic architecture pattern

A CNN block typically plays three roles:

1. **Convolution** — extract local patterns
2. **Activation** — add nonlinearity to learn complex relations
3. **Downsampling** — reduce resolution, grow the field of view, cut compute

Not every block has all three, but it's the recurring motif of classic CNNs: stacked several times, then a **FC head** at the end for classification:

```
[Conv → ReLU → Pool] × N  →  Flatten  →  FC → Softmax
```

Deeper in, spatial resolution shrinks (pooling) while the number of channels grows — trading "where" for "what".

## The CNN family before ResNet

| Year | Network | Architectural significance |
|---|---|---|
| 1998 | LeNet-5 | Showed conv + pooling + FC is a practical pipeline for image recognition |
| 2012 | AlexNet | Scaled CNNs with GPUs, ReLU, dropout; opened the deep learning era on ImageNet |
| 2014 | VGG | Standardized the pattern of stacking many small $3\times3$ convs instead of large kernels |
| 2014 | GoogLeNet / Inception | Multi-scale branches + $1\times1$ bottleneck to add width while controlling compute |

Inception left two ideas that still matter today. **Multi-scale**: instead of picking one kernel size, an Inception block runs several kernels in parallel ($1\times1$, $3\times3$, $5\times5$) and concatenates the results — letting the network choose the right scale. **The $1\times1$ conv** (introduced above): used to *reduce the channel count* before expensive convs — exactly the idea ResNet reuses for its **bottleneck block**.

## Limitations of CNNs

CNNs are powerful when data has local spatial structure, like images. But that same locality bias is also a limitation:

- **Hard to train when too deep**: the pre-2015 trend was to add depth (LeNet a few layers → AlexNet 8 → VGG 16–19) for more abstract features. But deeper is *not automatically* better — too deep and training error rises due to **degradation**.
- **Long-range relations need many layers** to connect (the receptive field only grows gradually).
- **Fixed square kernels** are inflexible for long-range structure or global context.
- **Pooling/downsampling** can discard spatial detail.

These limitations motivated later designs: **ResNet** (residual connections to train very deep nets), then FPN / dilated convs (multi-scale), and finally **attention / Vision Transformers** (direct global relations).
